# Hackathon Pitch Strategy

Guidelines for constructing and delivering an effective hackathon pitch that persuades judges within strict time constraints.

---

## Pitch Architecture

Every effective hackathon pitch follows this structure:

```
[Hook]          → Grab attention in 15 seconds
[Problem]       → Make judges feel the pain
[Solution]      → Reveal the answer
[Demo]          → Show it working
[Technology]    → Establish credibility
[Impact]        → Scale the vision
[Team]          → Why you will execute
[Call to Action]→ Leave judges with something to remember
```

---

## Time Allocation by Pitch Length

| Segment        | 2-min pitch | 3-min pitch | 5-min pitch |
| -------------- | ----------- | ----------- | ----------- |
| Hook + Problem | 20s         | 30s         | 40s         |
| Solution       | 15s         | 20s         | 30s         |
| Demo           | 50s         | 80s         | 120s        |
| Technology     | 10s         | 15s         | 20s         |
| Impact         | 10s         | 15s         | 20s         |
| Team           | 5s          | 10s         | 15s         |
| Call to Action | 10s         | 10s         | 15s         |

---

## Hook Patterns

### Pattern 1: The Statistic Hook

> "Every year, X billion dollars / X million people / X% of [group] suffer from [problem]."

Use only if the statistic is verifiable and viscerally large.

### Pattern 2: The Story Hook

> "Last [time period], [persona] faced [situation]. They [action]. It didn't work. Here's why."

Most effective for relatable consumer or professional problems.

### Pattern 3: The Question Hook

> "What if [existing broken thing] actually [worked the way you wanted]?"

Effective for tools and workflow improvements.

### Pattern 4: The Demonstration Hook

Start by showing the product working — no introduction.

Most effective when the product is visually stunning or immediately comprehensible.

---

## Problem Framing Rules

1. The problem must be understandable without domain expertise.
2. Quantify the pain: time lost, money lost, people affected, or frequency of occurrence.
3. Explain why existing solutions fail — not just that they fail.
4. Make at least one judge nod in recognition.

---

## Solution Framing Rules

1. State the solution in one sentence before showing it.
2. Use the phrase "so that [outcome]" — connect mechanism to benefit.
3. Do not explain how it works before showing that it works.
4. Name the product clearly; judges forget unnamed projects.

---

## Demo Integration

- Transition with: "Let me show you."
- Narrate outcomes, not actions: "Now [persona] can..." not "I'm clicking..."
- Return from the demo with: "What you just saw was..." — reinforce the value.

---

## Technology Slide Principles

- One slide maximum.
- List technologies, not explanations.
- Highlight what's novel: "We built X because no existing library does Y."
- Mention sponsor API or prize track technology explicitly.

---

## Q&A Strategy

| Question Type                    | Response Strategy                                                                        |
| -------------------------------- | ---------------------------------------------------------------------------------------- |
| "Why not just use [competitor]?" | Acknowledge, then differentiate: "Great point — [X] does Y, but we do Z"                 |
| "How does it scale?"             | Acknowledge limits: "Today we handle X; next step is Y"                                  |
| "What's the business model?"     | Frame as validation-first: "We're solving the problem first; monetization is [approach]" |
| "Is this technically feasible?"  | Point to the demo: "It's working now; here's what's real vs. mocked"                     |
| "Why are you the right team?"    | Lived experience + complementary skills                                                  |
| Questions you don't know         | "We don't know yet — here's how we'd find out"                                           |

---

## Common Pitch Failures

| Failure                        | Fix                                                |
| ------------------------------ | -------------------------------------------------- |
| Starting with team intro       | Start with the hook or problem                     |
| Feature list instead of story  | Build a narrative arc                              |
| Demo longer than 50% of pitch  | Cut secondary features                             |
| Apologizing for what's missing | Emphasize what works                               |
| Monotone delivery              | Vary pace; slow at the wow moment                  |
| No clear ask or close          | End with a memorable statement or question         |
| Going over time                | Practice to 80% of limit; buffer for live demo lag |
