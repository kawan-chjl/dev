# Hackathon Demo Patterns

Proven demo formats and presentation techniques that create strong impact during hackathon judging.

---

## Demo Format Patterns

### Pattern 1: The Problem → Solution Arc

**Structure:** Show the broken world → show your world → contrast.

1. Open with a painful scenario without your product.
2. Say "with [project name]..." and transition to the demo.
3. Complete the same scenario in a fraction of the time/effort.

**Best for:** Consumer apps, productivity tools, workflow automation.

---

### Pattern 2: The Live Magic Moment

**Structure:** Build tension → execute the impressive action → let silence land.

1. Set up the input or context.
2. Trigger the main capability (inference, generation, integration).
3. Pause. Let the result speak. Then explain what just happened.

**Best for:** AI/ML features, real-time data, generative outputs.

---

### Pattern 3: The Persona Walk-Through

**Structure:** Play the role of the target user navigating the product.

1. Introduce the persona: "I'm a nurse in a rural hospital..."
2. Walk through a realistic scenario step by step.
3. Show the outcome and state the value delivered.

**Best for:** Domain-specific tools, healthcare, fintech, education.

---

### Pattern 4: The Data Before/After

**Structure:** Show the problem as data → transform it → show the result as data.

1. Display messy, incomplete, or broken input data.
2. Run the pipeline/model/process.
3. Display clean, enriched, or actionable output.

**Best for:** Data engineering, analytics, ML pipelines, API products.

---

### Pattern 5: The Side-by-Side Comparison

**Structure:** Old way on left, your way on right. Same task. Show time delta.

- Use a timer or explicit step count.
- Keep both paths visible simultaneously if possible.
- Quantify the improvement in one sentence.

**Best for:** Developer tools, automation, accessibility features.

---

## Demo Delivery Techniques

### Narration Principles

- One sentence per screen state. Never narrate what the judge can see.
- Use present tense: "The system is analyzing..." not "As you can see, it analyzed..."
- Slow down before the wow moment; speed up during setup.

### Visual Principles

- Zoom or highlight the key element before clicking.
- Use a large mouse cursor; zoom to 125–150% on small screens.
- Dark mode reduces glare and looks better on projectors.

### Reliability Principles

- Use pre-seeded test data; never type live inputs from scratch.
- Have a fallback recording for each critical demo step.
- Run the demo on the machine that will be used for judging, not a different device.

## Demo Anti-Patterns

| Anti-Pattern                         | Why It Fails                                     |
| ------------------------------------ | ------------------------------------------------ |
| Showing the code                     | Judges want the product, not the implementation  |
| Too many features                    | Dilutes the wow moment; judges remember nothing  |
| Apologizing for bugs                 | Destroys confidence; pivot or skip gracefully    |
| Reading slides aloud                 | Judges can read; narrate the story instead       |
| Long loading times without narration | Kills momentum; narrate through every wait state |
| Asking "can you see my screen?"      | Breaks the narrative flow                        |
| Ending on a thank-you slide          | Ends on low energy; close with the vision or CTA |

## Demo Length Benchmarks

| Event Type       | Ideal Demo Duration | Notes                           |
| ---------------- | ------------------- | ------------------------------- |
| 3-minute pitch   | 60–90 seconds       | Leave time for problem and team |
| 5-minute pitch   | 90–120 seconds      | Can show 2–3 features           |
| Expo table       | 60–90 seconds       | Must be repeatable 10+ times    |
| Video submission | Full video length   | One complete arc                |
