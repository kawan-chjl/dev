# Hackathon Submission Guidelines

Best practices for writing, assembling, and submitting a hackathon project that scores well on both automated platform review and human judge evaluation.

---

## Submission Completeness Checklist

Every hackathon submission should include all of the following:

| Artifact                         | Required    | Notes                         |
| -------------------------------- | ----------- | ----------------------------- |
| Project title                    | Yes         | Memorable, ≤5 words preferred |
| Tagline / short description      | Yes         | ≤280 characters               |
| Full project description         | Yes         | See structure below           |
| Demo video                       | Yes         | Most platforms require this   |
| Source code repository           | Yes         | Must be public or shared      |
| Live demo URL                    | Recommended | Dramatically increases scores |
| Team member profiles             | Yes         | Include roles                 |
| Track / prize category selection | Yes         | Select all applicable         |
| Tech stack tags                  | Recommended | Used for filtering            |
| Screenshots                      | Recommended | ≥3 screenshots of key screens |

---

## Project Description Structure

Write the submission description in this order:

### 1. Inspiration (2–3 sentences)

- What problem inspired the project?
- Who is affected and how severely?
- Why did your team care about this problem?

### 2. What It Does (3–5 sentences)

- What does the product do from the user's perspective?
- What is the primary user journey?
- What is the key output or outcome for the user?

### 3. How We Built It (3–5 sentences)

- Architecture overview.
- Key technologies and APIs used.
- Any novel technical decisions worth highlighting.

### 4. Challenges We Ran Into (2–3 sentences)

- Honest description of the hardest technical or product challenge.
- Shows judges that the problem was genuinely hard.
- Demonstrates team problem-solving ability.

### 5. Accomplishments We're Proud Of (2–3 bullets)

- Specific, concrete achievements.
- Prefer technical accomplishments for technical judge panels.
- Prefer user-centered accomplishments for business/impact tracks.

### 6. What We Learned (2–3 sentences)

- Technical insights.
- Domain insights.
- Team process insights.

### 7. What's Next (2–3 bullets)

- Near-term roadmap items.
- Validation experiments planned.
- Connection to the 10x vision.

---

## Writing Quality Rules

1. Use active voice throughout.
2. Avoid jargon that requires domain expertise to understand.
3. Every claim must be verifiable from the demo or the repository.
4. Do not use placeholder text; every section must be completed.
5. Proofread for typos — judges notice.
6. Keep paragraphs to 3 sentences maximum.

---

## Repository README Requirements

The repository README must include:

- Project name and one-sentence description
- Screenshot or GIF of the product in action
- Tech stack list
- Setup instructions (one-command preferred)
- Link to live demo (if available)
- Link to demo video
- Team members

---

## Demo Video Requirements

| Requirement | Guideline                               |
| ----------- | --------------------------------------- |
| Length      | 90 seconds to 3 minutes                 |
| Format      | MP4 preferred; minimum 720p             |
| Content     | Problem → Solution → Demo → Close       |
| Audio       | Clear narration; no background noise    |
| Captions    | Strongly recommended                    |
| Branding    | Project name visible in first 3 seconds |

---

## Platform-Specific Tips

### Devpost

- Use all available tag fields for discoverability.
- Add a gallery with ≥3 images before the video.
- Enable "Open Source" flag if the repo is public.

### Dorahacks

- BUIDL description is public and searchable.
- Add team member GitHub profiles.
- Include a demo GIF in the gallery.

### HackerEarth

- Tech stack tagging affects filtering in judge dashboards.
- Submit at least 2 hours before deadline; servers slow down.

---

## Submission Timing

| Action                      | When                              |
| --------------------------- | --------------------------------- |
| Create submission draft     | Within first 4 hours of hackathon |
| Fill in all required fields | At least 2 hours before deadline  |
| Upload demo video           | At least 1 hour before deadline   |
| Final review and submit     | 30–60 minutes before deadline     |
| Confirm submission email    | Immediately after submitting      |

**Never submit in the final 15 minutes.** Platform traffic spikes cause submission failures close to the deadline.
